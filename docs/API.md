# API 명세

> **이 문서는 언제 작성하나?**
> 서버에서 데이터를 가져오는 **자체 API 엔드포인트가 필요한 규모 있는 프로젝트**일 때만 작성한다.
> (정적/UI-only/소규모 프로젝트는 불필요 — 이 파일을 두지 않는다.)
>
> **순서가 중요하다.** API가 필요한 프로젝트는 이 명세를 **먼저 작성하고 검증**한 뒤에 harness를 실행한다.
> 엔드포인트·스키마가 확정되지 않은 채 구현을 시작하면 step 간 인터페이스가 어긋난다.
> 검증 = 아래 "검증 체크리스트"를 모두 통과.

## 공통 규약

### 에러 포맷
모든 에러 응답은 동일한 형태를 따른다:
```json
{ "error": { "code": "STRING_CODE", "message": "사람이 읽을 설명" } }
```

### 상태 코드 컨벤션
| 코드 | 의미 |
|------|------|
| 200 / 201 | 성공 / 생성됨 |
| 400 | 잘못된 요청(검증 실패) |
| 401 / 403 | 인증 필요 / 권한 없음 |
| 404 | 리소스 없음 |
| 500 | 서버 오류 |

### 페이지네이션 (목록 엔드포인트)
{예: `?page=1&limit=20`, 응답에 `{ items, total, page, limit }`}

## 엔드포인트 명세

각 엔드포인트를 아래 형식으로 모두 기술한다.

### `{METHOD} {경로}` — {한 줄 설명}
- **인증**: {필요/불필요}
- **요청**:
  - Path params: {`id: string` 등 / 없음}
  - Query: {`q?: string` 등 / 없음}
  - Body:
    ```json
    { "field": "type — 설명" }
    ```
- **응답 (200)**:
  ```json
  { "field": "type — 설명" }
  ```
- **에러**: {발생 가능한 상태코드와 code}
- **예시**:
  ```
  GET /api/items?limit=2
  → 200 { "items": [...], "total": 57, "page": 1, "limit": 2 }
  ```

## 데이터 모델

핵심 엔티티의 타입을 정의한다 (구현의 `src/types`와 일치시킨다).

```ts
  id: string;
  // ...
};
```

## 검증 체크리스트 (harness 실행 전 통과 필수)

- [ ] 모든 엔드포인트에 method/경로/요청/응답/에러가 빠짐없이 기술됨
- [ ] 요청·응답 스키마가 `데이터 모델`의 타입과 일치
- [ ] 에러 포맷·상태코드가 `공통 규약`을 따름
- [ ] 프론트엔드가 호출할 모든 데이터가 어떤 엔드포인트에서 오는지 커버됨 (누락된 데이터 흐름 없음)
- [ ] 인증이 필요한 엔드포인트가 명시됨
